## Принцип работы балансировки задач - Round Robin.

Счётчик roundRobinCounter - атомарный (AtomicInteger), чтобы два
потока-отправителя не попали в одну и ту же очередь из-за гонки.

Если выбранная очередь заполнена - алгоритм перебирает остальные очереди
по кругу. Если все заняты - инициируется политика отказа.

Алгоритм Least Loaded более справедлив, но имеет сложность O(n), а у Round Robin O(1). Плюс Round Robin проще в реализации и обеспечивает более сбалансированную нагрузку.

## Политика отказа

Я реализовала три политики:

AbortPolicy - просто выбрасывает исключение. Сразу видно, что пошло не так, но нужно обязательно поймать исключение или поток-отправитель упадёт.

DiscardPolicy - "молча" выкидывает задачу, не блокируя поток, но задача просто теряется. Удобно для малозначимых задач вроде метрик.

CallerRunsPolicy - отправитель сам выполняет задачу, если её не может выполнить целевой поток. Задача не теряется, но поток-отправитель фактически блокируется.

Я выбрала CallerRunsPolicy как решение по умолчанию т.к.:

1. Задача не теряется - это важно для корректности системы.
2. Создаётся обратное давление (backpressure): пока поток-отправитель
   занят выполнением задачи сам, он не присылает новых задач. Пул получает
   передышку. 
3. Недостаток: если отправитель - main-поток или UI-поток, он зависнет.
   В таких случаях лучше AbortPolicy с обработкой исключения.

## Анализ производительности

### Сравнение с JDK ThreadPoolExecutor

Распределение задач: в CustomThreadPool - Per-thread queues (снижает конкуренцию за лок) + Round Robin. В ThreadPoolExecutor - одна общая очередь.
Создание потоков: в CustomThreadPool - при запуске (core) + по требованию. В ThreadPoolExecutor - при запуске (core) + по требованию.
Синхронизация: в CustomThreadPool - Глобальный lock для изменений. В ThreadPoolExecutor - Оптимизированный ReentrantLock (более тонкая и надёжная синхронизация).

### Сравнение с Tomcat

Tomcat использует ThreadPoolExecutor с собственными модификациями. Кастомная очередь TaskQueue отдаёт предпочтение созданию потоков перед постановкой в очередь - это снижает латентность при всплесках нагрузки. 

Мой пул реализует схожую идею через minSpareThreads - всегда есть резервные потоки. Но у меня per-thread очереди с Round Robin, у Tomcat одна общая очередь. Т.е. в моём пуле меньше конкуренция за очередь, но хуже балансировка неравномерных по времени задач. А у Tomcat такой проблемы нет. 

### Сравнение с Jetty

Jetty использует QueuedThreadPool со схожей идеей. 

Jetty использует QueuedThreadPool - одна очередь, но с настраиваемым min/maxThreads и reservedThreads (minSpareThreads - мой аналог). Но Jetty использует work-stealing — свободный поток может "украсть" задачу из очереди другого. Это устраняет проблему застрявшей очереди, которая может быть у моей системы.

## Исследование оптимальных параметров

### corePoolSize

- Слишком мало (1-2): при всплеске нагрузки очереди забиваются,
  латентность растёт.
- Слишком много: потоки "крутятся вхолостую", тратят память (~1MB/поток)
  и CPU на context switching.
- Оптимум: для CPU-bound задач - N_CPU + 1,
  для IO-bound (сетевые запросы, БД) - N_CPU * 2 или больше
  (потоки большую часть времени ждут IO).

### queueSize

- Слишком маленький: частые отказы при кратковременных всплесках.
- Слишком большой: большая задержка для "хвоста очереди",
  высокое потребление памяти, backpressure не работает вовремя.
- Оптимум: для latency-sensitive систем - небольшой (10-100),
  с CallerRunsPolicy как backpressure-механизмом.

### keepAliveTime

- Короткий (< 1 сек): потоки часто создаются/удаляются - overhead.
- Длинный (> 60 сек): лишние потоки долго висят вхолостую.
- Оптимум для большинства сервисов: 5-30 секунд.

### minSpareThreads

- 0: потоки создаются только под нагрузку (экономно, но медленный старт).
- больше 0: быстрая реакция на всплески, ценой фонового потребления.
- Для HTTP-серверов обычно 1-5.
